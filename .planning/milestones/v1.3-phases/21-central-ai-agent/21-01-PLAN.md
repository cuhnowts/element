**mcp-server/src/db.ts:**
- Import `Database` from `better-sqlite3`
- Accept DB path from `process.argv[2]` (passed by Tauri at launch)
- If no arg provided, check `process.env.ELEMENT_DB_PATH` as fallback
- If neither, print error to stderr and exit(1)
- Open with `{ readonly: true }` per research pitfall 1
- Run `PRAGMA journal_mode=WAL;` and `PRAGMA busy_timeout=5000;` for safe concurrent access
- Export the `db` instance

**mcp-server/src/tools/project-tools.ts:**
- Export function `handleListProjects(db)`: query `SELECT id, name, description, directory_path, planning_tier, created_at, updated_at FROM projects ORDER BY created_at DESC`, return as JSON text content
- Export function `handleGetProjectDetail(db, { projectId })`: query single project by id, include phases count and task count via subqueries. Return 404-style error text if not found.

**mcp-server/src/tools/phase-tools.ts:**
- Export function `handleListPhases(db, { projectId })`: query `SELECT id, project_id, name, description, sort_order, source, created_at FROM phases WHERE project_id = ? ORDER BY sort_order ASC`
- Export function `handleGetPhaseStatus(db, { projectId })`: for each phase, count total tasks and completed tasks (status = 'done'). Return phases with `{ id, name, totalTasks, completedTasks, isComplete: completedTasks === totalTasks && totalTasks > 0 }`

**mcp-server/src/tools/task-tools.ts:**
- Export function `handleListTasks(db, { projectId, phaseId? })`: query tasks with optional phase filter. Fields: `id, title, description, status, priority, phase_id, estimated_minutes, due_date`

**mcp-server/src/tools/orchestration-tools.ts:**
The approval flow uses a file-based queue per research recommendation (open question 1, option a). The queue directory is `<db_dir>/agent-queue/`.

- `handleRequestApproval(db, { projectId, projectName, phaseName, reason })`:
  - Generate unique approval ID: `approval-<timestamp>-<random4chars>`
  - Write JSON file to `<queue_dir>/approvals/<approvalId>.json` containing: `{ id, projectId, projectName, phaseName, reason, status: "pending", createdAt: ISO string }`
  - Return the approval ID as text content

- `handleCheckApprovalStatus(db, { approvalId })`:
  - Read `<queue_dir>/approvals/<approvalId>.json`
  - Return `{ id, status, decidedAt? }` -- status is "pending", "approved", or "rejected"
  - If file not found, return error text "Approval not found"

- `handleSendNotification(db, { title, body, priority, projectId?, projectName? })`:
  - Write JSON file to `<queue_dir>/notifications/<timestamp>-<random>.json` containing the notification payload
  - Priority must be one of: "critical", "informational", "silent"
  - Return confirmation text

- `handleReportStatus(db, { action, status, projectId?, details? })`:
  - Write JSON file to `<queue_dir>/status/<timestamp>-<random>.json`
  - Status must be one of: "completed", "failed", "blocked"
  - Return confirmation text

- `handleSpawnProjectSession(db, { projectId, sessionName })`:
  - Write JSON file to `<queue_dir>/sessions/<timestamp>-<random>.json` containing: `{ projectId, sessionName, createdAt }`
  - The frontend will watch this directory and spawn the actual terminal session
  - Return confirmation text with session request ID

For all orchestration tools, create the queue subdirectories (`approvals/`, `notifications/`, `status/`, `sessions/`) on first use with `mkdirSync({ recursive: true })`.
