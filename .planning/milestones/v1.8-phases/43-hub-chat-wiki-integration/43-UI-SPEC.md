## Component Inventory

This phase reuses existing components and extends one pattern. No new shadcn components needed.

### Reused As-Is

| Component | File | Usage in Phase 43 |
|-----------|------|--------------------|
| ActionConfirmCard | `src/components/hub/ActionConfirmCard.tsx` | Wiki ingest confirmation (D-08). Extend `ACTION_ICON_MAP`, `ACTION_TITLE_MAP`, `REJECT_LABEL_MAP`, and `getBodyText` with wiki-specific entries. |
| ActionResultCard | `src/components/hub/ActionResultCard.tsx` | Wiki operation success/failure results (D-09) |
| Card | `src/components/ui/card.tsx` | Base for all inline cards |
| Button | `src/components/ui/button.tsx` | Confirm/reject buttons |
| Badge | `src/components/ui/badge.tsx` | Only if destructive flag is set (not expected for wiki actions) |
| Separator | `src/components/ui/separator.tsx` | Divider in confirmation cards |

### New Entries in Existing Maps

ActionConfirmCard requires these additions to its static maps:

```
ACTION_ICON_MAP:
  wiki_ingest: BookPlus (from lucide-react)

ACTION_TITLE_MAP:
  wiki_ingest: "Add to Wiki"

REJECT_LABEL_MAP:
  wiki_ingest: "Don't Add"

getBodyText("wiki_ingest", input):
  return `Add this content to the wiki? A new article will be created in .knowledge/wiki/.`
```

### No New Components Needed

- Wiki query results render as normal bot markdown messages (ReactMarkdown with remarkGfm) — no special card.
- Wiki lint results render as ActionResultCard with the lint summary.
- In-progress status renders as existing bot message streaming pattern ("Searching wiki..." followed by the answer).
- Plugin tool definitions merge into the existing tool pipeline — no UI for tool management.

## Interaction Contracts

### Wiki Ingest Flow (D-06, D-08)

1. User pastes content and says "add this to the wiki"
2. LLM emits `tool_use` with `wiki_ingest` tool name
3. `useActionDispatch` detects `wiki_ingest` has `destructive: true` (per D-08, ingest writes to filesystem)
4. ActionConfirmCard renders with "Add to Wiki" title, BookPlus icon, approve/reject buttons
5. On approve: `dispatch_plugin_skill` Tauri command is invoked
6. ActionResultCard renders with success/failure message
7. On reject: "Don't Add" — card resolves as dismissed, bot responds naturally

### Wiki Query Flow (D-05, D-07)

1. User asks "what do I know about X?"
2. LLM emits `tool_use` with `wiki_query` tool name
3. `useActionDispatch` detects `wiki_query` has `destructive: false` — auto-executes (D-08)
4. `dispatch_plugin_skill` Tauri command is invoked
5. Result returned to LLM as tool_result
6. LLM synthesizes answer inline as a normal bot message with markdown citations

### Wiki Lint Flow

1. User asks "check the wiki for issues"
2. LLM emits `tool_use` with `wiki_lint` tool name
3. Auto-executes (read-only, D-08)
4. ActionResultCard renders with lint summary

### Dynamic Tool Loading (D-01, D-02)

1. On HubChat mount, fetch plugin skills via `getPluginToolDefinitions()` (new function)
2. Merge with `getToolDefinitions()` from ACTION_REGISTRY
3. Pass merged tool list to `hub_chat_send` (already accepts `tools: Option<Vec<ToolDefinition>>`)
4. `buildSystemPrompt()` dynamically generates "Available Tools" section from both registries (D-04)
5. No UI indicator for loaded plugin tools — they appear transparently in AI capabilities

### System Prompt Assembly (D-03, D-04)

1. `buildSystemPrompt()` refactored from hardcoded tool list to dynamic assembly
2. Core tools section generated from `ACTION_REGISTRY`
3. Plugin tools section generated from `getPluginToolDefinitions()` result
4. Both sections concatenated into "Available Tools" in system prompt
5. All tools included in every call (D-03) — no filtering at MVP scale
