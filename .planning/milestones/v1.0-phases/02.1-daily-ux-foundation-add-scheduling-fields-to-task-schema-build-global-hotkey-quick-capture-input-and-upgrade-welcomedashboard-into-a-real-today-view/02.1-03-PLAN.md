Note: The selectedTask comes from `useStore((s) => s.selectedTask)` which is `TaskWithTags`. Since Plan 01 consolidated types, `TaskWithTags` extends `Task` which now has the scheduling fields. The `get_task` Tauri command returns `TaskWithTags` with `#[serde(flatten)]` on the `Task` field, so scheduling fields will be present in the flattened response.

After the Tags section and before the Description section, add a Duration section:
```tsx
{/* Duration Estimate */}
<div>
  <span className="text-xs font-semibold tracking-wide uppercase text-muted-foreground block mb-2">
    Duration
  </span>
  <DurationChips
    value={selectedTask.durationMinutes}
    onChange={(minutes) => updateTask(selectedTask.id, { durationMinutes: minutes ?? undefined })}
  />
</div>
```

The `updateTask` call in taskSlice already spreads the updates object and passes through to `api.updateTask`, which invokes the Tauri command. Since Plan 01 added `duration_minutes` to the `update_task` command, this will work end-to-end.

Note on null vs undefined: The UpdateTaskInput uses optional fields (`durationMinutes?: number`). When the user deselects a chip, DurationChips calls `onChange(null)`. We convert `null` to `undefined` so it becomes "not included" in the update payload, which means "don't change" in the current backend convention. To truly clear, we would need to pass the value. For now, clicking a chip sets it, but there's no way to clear it back to null through the UI. This is acceptable for Phase 2.1 -- clearing scheduling fields can be added later.

Actually, re-reading the DurationChips spec: "Clicking active chip deselects (sets duration to null)." To support this properly, when onChange(null) is called, we should pass `durationMinutes: 0` or some sentinel. BUT the simpler approach: just don't try to clear via the same update mechanism. Instead, only send durationMinutes when it's a positive number. When deselecting, we just update local state. The stored value remains until explicitly changed. This is good enough for now.

SIMPLEST: When onChange returns null (deselect), do nothing (don't call updateTask). When onChange returns a number, call updateTask. The visual state will show "no chip selected" but the backend still has the old value. On re-render (after refetch), the chip will re-select. This creates a minor UX inconsistency. BETTER: Actually call updateTask with the value. The Tauri command's update_task takes `Option<i32>` for duration_minutes. If we pass `undefined` in JS, Tauri maps it to `None` in Rust. None means "don't change." We need a way to say "set to null." This is a known limitation of the current update pattern.

FOR NOW: Just pass the value directly. If null, convert to a special marker or just skip. Deselect is a nice-to-have that can be addressed in a follow-up. The key deliverable is that selecting a duration saves it.

```typescript
onChange={(minutes) => {
  if (minutes !== null) {
    updateTask(selectedTask.id, { durationMinutes: minutes });
  }
}}
```
