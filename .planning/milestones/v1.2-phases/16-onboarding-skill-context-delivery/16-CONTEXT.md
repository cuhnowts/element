# Phase 16: Onboarding Skill and Context Delivery - Context

**Gathered:** 2026-03-27
**Status:** Ready for planning

<domain>
## Phase Boundary

Add a product knowledge section to the AI context file so the AI assistant understands what Element is, its core concepts, and its role within the app. The skill content is delivered as a new section at the top of `.element/context.md`, generated alongside existing project context on every "Open AI" click.

**This phase does NOT:**
- Change the tier selection flow (Phase 14)
- Add .planning/ sync (Phase 15)
- Change the context file structure for project data (Phase 13)

</domain>

<decisions>
## Implementation Decisions

### Content Scope
- **D-01:** Orientation brief — explains what Element is, its core concepts (themes, projects, phases, tasks), and the AI's role within the app. Target ~300-500 tokens.
- **D-02:** Includes AI role framing — tells the AI it's working inside Element, that it's been seeded with project context, and what actions it can take (help plan, work on tasks, answer questions about the project).
- **D-03:** Semi-dynamic content — static product description hardcoded in Rust, plus dynamic injection of the configured CLI tool name and the project's planning tier. No extra DB queries beyond what's already available in the generation function.

### Delivery Mechanism
- **D-04:** New `## About Element` section added at the top of the existing `context.md`, before the project Header section. Generated by a new function in `onboarding.rs`.
- **D-05:** Single file delivery — skill content is part of the same `.element/context.md` file, not a separate file. One generation path, one file passed to the CLI tool.

### Tier/State Awareness
- **D-06:** Same product knowledge for all tiers (Quick, Medium, GSD). The skill section does not change based on tier — the existing Instructions section already handles tier-specific behavior.
- **D-07:** Always present regardless of project state (NoPlan, Planned, InProgress, Complete). AI always gets product orientation.

### Token Budget
- **D-08:** Separate budget — the skill section has its own ~400 token allowance on top of the existing ~2000 token project data budget. Product knowledge does not compete with task/phase data. Total context grows to ~2400 tokens max.
- **D-09:** Skill section always renders in full — no progressive collapse applied to it. Only project data gets collapsed under budget pressure.

### Content Maintenance
- **D-10:** Hardcoded in Rust in `onboarding.rs` as a dedicated function (e.g., `build_skill_section`). Updated when new features ship, versioned with the app. No external template files or database storage.

### GSD Integration
- **D-11:** Skill section appears for GSD tier too. It's read-only orientation that doesn't conflict with GSD's own context gathering (.planning/CONTEXT.md, PROJECT.md). GSD users still benefit from knowing what Element is.

### Claude's Discretion
- Exact wording of the product description and AI role framing
- How to format the dynamic CLI tool name and tier within the section
- Whether to use a heading hierarchy (## About Element > ### Core Concepts) or flat structure

</decisions>
