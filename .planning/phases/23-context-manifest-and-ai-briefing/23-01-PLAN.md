2. **`generate_briefing` Tauri command** (per D-07, D-08):
```rust
#[tauri::command]
pub async fn generate_briefing(
    app: AppHandle,
    db_state: State<'_, Arc<Mutex<Database>>>,
    manifest_state: State<'_, ManifestState>,
    gateway: State<'_, AiGateway>,
) -> Result<(), String>
```
- Read cached manifest from `manifest_state.cached.lock()` (clone, drop lock)
- If manifest is empty, call `build_manifest_string` first (same pattern as build_context_manifest)
- Get provider config: `{ let db = db_state.lock()...; gateway.get_default_config(&db)... }` -- drop DB lock before async
- If no default provider, emit `app.emit("briefing-error", "No AI provider configured. Add one in Settings to enable the daily briefing.")` and return Ok(())
- Build provider: `gateway.build_provider(&provider_config)`
- Build CompletionRequest with:
  - `system_prompt`: A briefing system prompt (see below)
  - `user_message`: The cached manifest string
  - `max_tokens`: 1024
  - `temperature`: 0.7
- Create `tokio::sync::mpsc::channel::<String>(32)`
- Spawn forwarder task: `while let Some(chunk) = rx.recv().await { app_clone.emit("briefing-chunk", &chunk); }`
- Call `provider.complete_stream(request, tx).await`
- Wait for forwarder
- On Ok: emit `app.emit("briefing-complete", ())`
- On Err: emit `app.emit("briefing-error", e.to_string())`

    let hour = chrono::Local::now().hour();
    let time_context = if hour < 12 {

        - Flags any projects with no progress or many incomplete tasks\n\
        - Suggests what to focus on next\n\
        - Uses an encouraging but direct tone\n\

        Format your response as markdown. Use **bold** for project names and bullet points for action items.\n\
        Do NOT include a greeting -- the app adds one separately.\n\
        Do NOT repeat the raw project data back -- synthesize and prioritize.\n\
        Keep the response under 200 words.",
        time_context
    )
}

**D. Debounced manifest rebuild** (per D-05, CTX-03):

Add a `spawn_manifest_rebuilder` function in `manifest_commands.rs`:
```rust
pub fn spawn_manifest_rebuilder(app_handle: AppHandle) -> tokio::sync::mpsc::Sender<()>
```
- Creates a `tokio::sync::mpsc::channel::<()>(16)`
- Spawns a tokio task that:
  - Receives messages on the channel
  - After receiving, waits 5 seconds (`tokio::time::sleep(Duration::from_secs(5))`)
  - Drains any additional messages that arrived during the wait (non-blocking `try_recv` loop)
  - Locks DB from app_handle managed state, calls `build_manifest_string`, updates ManifestState cache
- Returns the sender

In `lib.rs` setup, after managing ManifestState:
```rust
let rebuild_tx = commands::manifest_commands::spawn_manifest_rebuilder(app.handle().clone());
app.manage(ManifestRebuildTrigger(rebuild_tx));
```

Add `ManifestRebuildTrigger` as a simple newtype wrapper:
```rust
pub struct ManifestRebuildTrigger(pub tokio::sync::mpsc::Sender<()>);
```
(Put this in manifest.rs alongside ManifestState)

Then in existing mutation commands (update_task_status, update_phase, create_task, delete_task), add a `rebuild_trigger.0.send(()).await` call. Specifically modify these files to accept the trigger state and send a unit message:
- `src-tauri/src/commands/task_commands.rs`: in `update_task_status`, `create_task`, `delete_task`
- `src-tauri/src/commands/phase_commands.rs`: in `update_phase`, `create_phase`, `delete_phase`

For each, add `rebuild_trigger: State<'_, ManifestRebuildTrigger>` parameter and `let _ = rebuild_trigger.0.try_send(());` after the DB mutation succeeds. Use `try_send` (non-blocking) so it never blocks the mutation command.
