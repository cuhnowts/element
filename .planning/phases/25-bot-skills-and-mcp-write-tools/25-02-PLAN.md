    **Create `mcp-server/src/tools/write-tools.ts`** with handlers following the existing pattern (function receives db + args, returns `{ content: [...] }`):

    1. `handleCreateTask(db, dbPath, args: { title: string; projectId?: string; description?: string; priority?: string; phaseId?: string })`:
       - Generate UUID via `crypto.randomUUID()`
       - Set defaults: status = "todo", priority = args.priority ?? "medium"
       - INSERT INTO tasks (id, title, description, status, priority, project_id, phase_id, created_at, updated_at) VALUES (...)
       - Call `emitDataChanged(dbPath, "task-created")` after successful write
       - Return JSON `{ id, title, status, priority }`

    2. `handleUpdateTask(db, dbPath, args: { taskId: string; title?: string; description?: string; priority?: string })`:
       - Build dynamic SET clause from provided fields
       - UPDATE tasks SET ... WHERE id = taskId, also set updated_at = now
       - Call `emitDataChanged(dbPath, "task-updated")` after successful write
       - Return JSON `{ taskId, updated: true }`

    3. `handleUpdateTaskStatus(db, dbPath, args: { taskId: string; status: string })`:
       - Validate status is one of: "todo", "in_progress", "done", "cancelled"
       - UPDATE tasks SET status = ?, updated_at = ? WHERE id = ?
       - Call `emitDataChanged(dbPath, "task-updated")` after successful write
       - Return JSON `{ taskId, status }`

    4. `handleDeleteTask(db, dbPath, args: { taskId: string })`:
       - DELETE FROM tasks WHERE id = ?
       - Call `emitDataChanged(dbPath, "task-deleted")` after successful write
       - Return JSON `{ taskId, deleted: true }`

    5. `handleUpdatePhaseStatus(db, dbPath, args: { phaseId: string; status: string })`:
       - Note: phases table does not have a status column directly. Phase "status" is derived from task completion. For now, this tool reports the requested status change as a status message. If a status column exists, update it. Otherwise return a descriptive message.
       - Call `emitDataChanged(dbPath, "phase-updated")` after any write
       - Return JSON `{ phaseId, status, note: "Phase status is derived from task completion" }`

    6. `handleCreateProject(db, dbPath, args: { name: string; description?: string })`:
       - Generate UUID, INSERT INTO projects (id, name, description, created_at, updated_at) VALUES (...)
       - Call `emitDataChanged(dbPath, "project-created")` after successful write
       - Return JSON `{ id, name }`

    7. `handleCreateTheme(db, dbPath, args: { name: string; color: string })`:
       - Generate UUID, INSERT INTO themes (id, name, color, created_at, updated_at) VALUES (...)
       - Call `emitDataChanged(dbPath, "theme-created")` after successful write
       - Return JSON `{ id, name, color }`

    8. `handleCreateFile(db, dbPath, args: { path: string; content: string; projectId: string })`:
       - Look up project's linked directory from projects table (external_path or directory field)
       - If project has no linked directory, return error
       - Resolve full path: `path.join(projectDir, args.path)`
       - Ensure parent directory exists (fs.mkdirSync recursive)
       - Write file with fs.writeFileSync
       - Call `emitDataChanged(dbPath, "file-created")` after successful write
       - Return JSON `{ path: fullPath, created: true }`

    9. **Helper function `emitDataChanged(dbPath: string, event: string)`:**
       - Derive the agent queue notifications directory from dbPath: `path.join(path.dirname(dbPath), "agent-queue", "notifications")`
       - Write a JSON file with unique name (e.g., `data-changed-{timestamp}.json`) containing `{ type: "data-changed", event, timestamp: new Date().toISOString() }`
       - This follows the existing agent queue file-based notification pattern (per RESEARCH.md Pitfall 2). The frontend polls the agent queue every 2 seconds via useAgentQueue and will pick up these notifications to trigger a data refresh.
