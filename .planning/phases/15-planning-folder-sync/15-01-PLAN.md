2. **parse_roadmap(content: &str) -> Result<ParseResult, String>:**
   - Use `regex::Regex` for patterns:
     - Phase header: `^### Phase (\d+(?:\.\d+)?): (.+)$`
     - Goal: `^\*\*Goal\*\*\s*:\s*(.+)$`
     - Criterion: `^\s+\d+\.\s+\[([ x])\]\s+(.+)$`
     - Backlog: `^### Phase 999`
   - Parse line-by-line. When backlog marker hit, set in_backlog=true and skip all remaining lines (per D-04).
   - Track current_phase, in_success_criteria, sort_order counter.
   - Success criteria section starts when line contains `**Success Criteria**`.
   - Success criteria section ends when a non-indented non-empty line is hit (that is not a criterion).
   - Return Err("No phases found in ROADMAP.md") if phases vec is empty.

3. **compute_content_hash(content: &str) -> String:**
   - Use `sha2::{Digest, Sha256}` (already in Cargo.toml).
   - Return lowercase hex string: `format!("{:x}", hasher.finalize())`.

4. **sync_roadmap_to_db(db: &Database, project_id: &str, result: &ParseResult) -> Result<SyncResult, String>:**
   - Open unchecked_transaction (follows batch_create_plan pattern).
   - DELETE FROM tasks WHERE project_id = ?1 AND source = 'sync'
   - DELETE FROM phases WHERE project_id = ?1 AND source = 'sync'
   - For each parsed phase: INSERT INTO phases with uuid::Uuid::new_v4(), source='sync', chrono::Utc::now().to_rfc3339(). Include phase description in a `description` column if it exists, otherwise store as part of the name (check schema -- phases table may not have description column; if not, append to name like "Name -- Goal text" or skip description for now since the UI only shows name).
   - For each task in phase: INSERT INTO tasks with source='sync', status based on is_complete ('complete' or 'pending'), priority='medium'.
   - tx.commit()
   - Return SyncResult with counts.

**Note on phase description:** Check if the phases table has a `description` column. The current Phase struct does NOT have a description field. Per D-03, the goal text becomes the phase description -- but since the schema doesn't support it, store the phase name as-is and the description is informational only (logged but not persisted). The tasks carry the real content. Alternatively, prepend goal to name if it's short, but this may clutter. Best approach: just store the phase name from the header. The goal text is implicit in the success criteria tasks.

1. **PlanningWatcherState struct:**
```rust
use std::sync::Mutex as StdMutex;

pub struct PlanningWatcherState {
    pub watcher: StdMutex<Option<notify_debouncer_mini::Debouncer<notify::RecommendedWatcher>>>,
    pub last_hash: StdMutex<Option<String>>,
}
```
The `last_hash` field stores the SHA-256 hash of the last synced ROADMAP.md content (in-memory, per Claude's discretion). This prevents re-sync when content hasn't changed (per D-08).

2. **sync_planning_roadmap command** (the core sync operation):
```rust
#[tauri::command]
pub async fn sync_planning_roadmap(
    app: AppHandle,
    db_state: State<'_, Arc<StdMutex<Database>>>,
    watcher_state: State<'_, PlanningWatcherState>,
    project_id: String,
    directory_path: String,
) -> Result<SyncResult, String>
```
- Read `.planning/ROADMAP.md` from `directory_path + "/.planning/ROADMAP.md"`.
- Compute SHA-256 hash via `compute_content_hash`.
- Compare with `watcher_state.last_hash`. If identical, return SyncResult { phase_count: 0, task_count: 0 } (silent skip, no toast -- per D-08).
- Parse via `parse_roadmap`. On Err, emit `planning-sync-error` event with error message string, return Err (per D-09).
- Lock DB, call `sync_roadmap_to_db`.
- Update `last_hash` with new hash.
- Emit `planning-sync-complete` event with `SyncResult` payload (per D-12).
- Return Ok(SyncResult).

3. **start_planning_watcher command:**
```rust
#[tauri::command]
pub async fn start_planning_watcher(
    app: AppHandle,
    watcher_state: State<'_, PlanningWatcherState>,
    db_state: State<'_, Arc<StdMutex<Database>>>,
    project_id: String,
    directory_path: String,
) -> Result<(), String>
```
- Build path: `PathBuf::from(&directory_path).join(".planning")`.
- If `.planning/` directory does not exist, return Ok(()) silently (no error, just nothing to watch).
- Create debouncer with 500ms interval (per Claude's discretion, matches existing patterns).
- In the debouncer callback:
  - Filter events for `ROADMAP.md` file_name (per D-06 + Pattern 2 from research: watch whole dir but only act on ROADMAP.md).
  - Read ROADMAP.md content.
  - Compute hash, compare with last_hash. If same, return early (no-op).
  - If different, emit `"planning-file-changed"` internal event with project_id payload so the frontend can call sync_planning_roadmap. OR -- simpler approach per research Pattern 5: do the full sync in the callback since DB lock is short-lived.
  - **Chosen approach:** In the callback, read file + compute hash + compare. If changed, use `app_handle.emit("planning-file-changed", &project_id)` to notify frontend. The frontend then calls `sync_planning_roadmap` command. This avoids holding DB lock in the notify thread (per Pitfall 4).
- Watch the `.planning/` directory with `RecursiveMode::NonRecursive` (per D-06 and anti-pattern note).
- Store debouncer in `watcher_state.watcher`.

4. **stop_planning_watcher command:**
```rust
#[tauri::command]
pub async fn stop_planning_watcher(
    watcher_state: State<'_, PlanningWatcherState>,
) -> Result<(), String>
```
- Lock `watcher_state.watcher`, set to None (drops debouncer, stops watching).
- Clear `watcher_state.last_hash` to None (fresh start on next project).
