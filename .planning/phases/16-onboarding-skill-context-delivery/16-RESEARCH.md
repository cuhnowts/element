## Summary

Phase 16 adds a product orientation section to the AI context file generated on every "Open AI" click. The implementation is entirely within the existing Rust context generation pipeline in `src-tauri/src/models/onboarding.rs` and its caller in `onboarding_commands.rs`. No new libraries, no new files, no frontend changes, no database schema changes.

The existing codebase follows a clear pattern: standalone `build_*` functions return `String`, composed sequentially in `generate_context_file_content()`. The new `build_skill_section()` follows this exact pattern. The only dynamic data needed (CLI tool name and planning tier) are already available in the generation pipeline or via `db.get_app_setting("cli_command")`.

**Primary recommendation:** Add `build_skill_section(cli_tool: &str, tier: &str) -> String` to `onboarding.rs`, call it first in `generate_context_file_content()`, and pass the CLI tool name from `generate_context_file` command where `db.get_app_setting("cli_command")` is already accessible.

## Architecture Patterns

### Existing Section Builder Pattern

Every context section follows this pattern in `onboarding.rs`:

```rust
fn build_section_name(params...) -> String {
    let mut out = String::from("## Section Title\n\n");
    // Build content with push_str / format!
    out.push('\n');
    out
}
```

Current composition order in `generate_context_file_content()`:
1. `build_header()` -- project name, description, tier, state, progress
2. `build_instructions()` -- tier+state instruction matrix
3. `build_attention_section()` -- in-progress/pending tasks (max 5)
4. `build_work_section()` -- phase/task rollup with token budget
5. `output_contract_section()` -- JSON output spec (Quick/Medium NoPlan only)

New composition order:
1. **`build_skill_section()` -- NEW, product orientation** (before header)
2. `build_header()` -- unchanged
3. `build_instructions()` -- unchanged
4. `build_attention_section()` -- unchanged
5. `build_work_section()` -- unchanged
6. `output_contract_section()` -- unchanged

### Signature Change to `generate_context_file_content`

Currently: `pub fn generate_context_file_content(data: &ProjectContextData, tier: &str) -> String`

The CLI tool name is not available in this function -- it's in the command handler where `db.get_app_setting("cli_command")` can be called. Two approaches:

**Option A (recommended):** Add `cli_tool: &str` parameter to `generate_context_file_content()`. The command handler already has DB access and fetches other settings. This keeps the model function pure (no DB dependency).

**Option B:** Add `cli_tool_name` field to `ProjectContextData`. More invasive since it changes the struct used by all callers.

Option A is cleaner -- one parameter addition, one query in the command handler.

### Anti-Patterns to Avoid
- **Putting DB calls in model functions:** `onboarding.rs` is a pure model file with no DB imports. Do not add `get_app_setting` here. Fetch in the command handler, pass as parameter.
- **Exceeding token budget:** The skill section targets ~400 tokens (~1600 chars). Keep the product description concise. Do not dump the full PROJECT.md content.
- **Tier-dependent skill content:** D-06 explicitly says same content for all tiers. Do not branch on tier within the skill section (tier is only mentioned as a dynamic label).

## Common Pitfalls

### Pitfall 1: Forgetting to update the function signature in tests
**What goes wrong:** Adding `cli_tool` parameter to `generate_context_file_content()` breaks all 20+ existing tests that call it with 2 parameters.
**Why it happens:** Test calls are not caught by the type system until compilation.
**How to avoid:** Update every `generate_context_file_content(&data, "tier")` call in tests to `generate_context_file_content(&data, "tier", "claude")` (or whatever the default is).
**Warning signs:** `cargo test` fails to compile immediately.

### Pitfall 2: CLI tool name being None
**What goes wrong:** `get_app_setting("cli_command")` returns `None` if user hasn't configured it yet.
**Why it happens:** New installs or users who haven't visited Settings.
**How to avoid:** Default to `"claude"` when the setting is missing: `cli_tool.unwrap_or("claude".to_string())`.
**Warning signs:** Skill section says "You're using (none)" or panics.

### Pitfall 3: Token count exceeding ~400 budget
**What goes wrong:** The skill section grows too large, crowding out project data in the AI context window.
**Why it happens:** Writing too much product description or explanation.
**How to avoid:** Keep the section to ~1600 characters. Write a test asserting `estimate_tokens(&build_skill_section("claude", "quick")) <= 500`.
**Warning signs:** Total context file exceeds ~2400 tokens for simple projects.

### Pitfall 4: Section ordering in output
**What goes wrong:** Skill section appears after the header instead of before it.
**Why it happens:** Adding the `push_str` call in the wrong position in `generate_context_file_content()`.
**How to avoid:** Add skill section as the FIRST `push_str` in the composition function, per D-04.
**Warning signs:** Context file starts with `# ProjectName` instead of `## About Element`.

## Code Examples

### build_skill_section pattern (verified from existing codebase patterns)

```rust
// Source: follows exact pattern of build_header(), build_instructions() in onboarding.rs
fn build_skill_section(cli_tool: &str, tier: &str) -> String {
    let mut out = String::from("## About Element\n\n");

    // Static product description (~200 tokens)
    out.push_str("Element is a desktop workflow orchestration platform — a personal work OS ");
    out.push_str("that organizes work into themes, projects, phases, and tasks. ");
    out.push_str("It structures your work top-down from high-level goals to actionable items.\n\n");

    // AI role framing (~100 tokens)
    out.push_str("### Your Role\n\n");
    out.push_str(&format!(
        "You are working inside Element, launched via `{}`. ", cli_tool
    ));
    out.push_str("This project's context has been seeded below. ");
    out.push_str(&format!(
        "The project uses the **{}** planning tier.\n\n", tier_display(tier)
    ));
    out.push_str("You can help by:\n");
    out.push_str("- Planning work (breaking down goals into phases and tasks)\n");
    out.push_str("- Working on tasks (writing code, documentation, configuration)\n");
    out.push_str("- Answering questions about the project's current state\n\n");

    out.push_str("---\n\n");
    out
}
```

pub fn generate_context_file_content(
    data: &ProjectContextData,
    tier: &str,
    cli_tool: &str

    let mut out = String::new();

    // Section 0: Skill/orientation (always present, D-07)
    out.push_str(&build_skill_section(cli_tool, tier));

    // Section 1: Header (existing)
    out.push_str(&build_header(data, tier, &state));
    // ... rest unchanged
}
