# Phase 02.1: Daily UX Foundation - Research

**Researched:** 2026-03-16
**Domain:** Task scheduling schema, global hotkey quick-capture, today view UI
**Confidence:** HIGH

## Summary

Phase 2.1 adds scheduling metadata to the task model (due date, scheduled date/time, duration, recurrence), builds a global-hotkey-triggered quick-capture floating window, and replaces the WelcomeDashboard with a time-aware Today View. The phase touches all layers: SQLite migration, Rust model/commands, TypeScript types, Zustand stores, and multiple new React components.

The technical stack is already established and well-understood. The primary complexities are: (1) Tauri multi-window management for the quick-capture floating window, including frameless/transparent window creation and cross-window state synchronization; (2) the global-shortcut plugin integration to trigger the capture window system-wide; and (3) the time-grouping logic for the Today View that must query across all projects and sort by scheduled time into fixed time-of-day buckets.

**Primary recommendation:** Build backend-first (migration, Rust models, commands), then Today View UI, then Quick-Capture window last since it has the most unknowns (multi-window, global shortcut registration, frameless window behavior on macOS).

### Pattern 1: SQLite Migration for New Columns

**What:** Add nullable columns to existing tasks table via ALTER TABLE statements in a new migration file.
**When:** Migration 003 runs automatically on app start when user_version < 3.

```sql
-- src-tauri/src/db/sql/003_scheduling.sql
ALTER TABLE tasks ADD COLUMN due_date TEXT;
ALTER TABLE tasks ADD COLUMN scheduled_date TEXT;
ALTER TABLE tasks ADD COLUMN scheduled_time TEXT;
ALTER TABLE tasks ADD COLUMN duration_minutes INTEGER;
ALTER TABLE tasks ADD COLUMN recurrence_rule TEXT
    CHECK(recurrence_rule IS NULL OR recurrence_rule IN ('daily', 'weekdays', 'weekly', 'monthly'));

CREATE INDEX IF NOT EXISTS idx_tasks_due_date ON tasks(due_date);
CREATE INDEX IF NOT EXISTS idx_tasks_scheduled_date ON tasks(scheduled_date);
```

**Why nullable:** All scheduling fields are optional per user decision. ALTER TABLE ADD COLUMN in SQLite requires columns to be nullable or have a DEFAULT -- nullable is correct here since empty means "no scheduling data."

**Migration runner update (migrations.rs):**
```rust
if version < 3 {
    conn.execute_batch(include_str!("sql/003_scheduling.sql"))?;
    conn.pragma_update(None, "user_version", 3)?;
}
```

### Pattern 3: Today View Time Grouping (Frontend Logic)

**What:** Group tasks into time-of-day buckets using scheduled_time and due_date.
**When:** TodayView renders task list.

```typescript
type TimeGroup = "overdue" | "morning" | "afternoon" | "evening" | "unscheduled";

function getTimeGroup(task: TodayTask, now: Date): TimeGroup {
  const today = now.toISOString().slice(0, 10);

  // Overdue: has due date before today AND not complete
  if (task.dueDate && task.dueDate < today && task.status !== "complete") {
    return "overdue";
  }

  // If task has scheduled time, use time-of-day buckets
  if (task.scheduledTime) {
    const hour = parseInt(task.scheduledTime.split(":")[0], 10);
    if (hour < 12) return "morning";
    if (hour < 17) return "afternoon";
    return "evening";
  }

  return "unscheduled";
}
```

### Pattern 4: Quick-Capture Multi-Window

**What:** Create a frameless, centered floating window from the main window's global shortcut handler.
**When:** User presses Ctrl+Space anywhere on the system.

The quick-capture window is a separate Tauri WebviewWindow that:
1. Loads a dedicated route/entry point (e.g., `/capture` or `capture.html`)
2. Shares the Rust backend state (Mutex<Database>) automatically -- all Tauri managed state is app-level, not window-level
3. Communicates back to the main window via Tauri events after task creation

```typescript
// Register global shortcut in main window on mount
import { register, unregister } from "@tauri-apps/plugin-global-shortcut";
import { WebviewWindow } from "@tauri-apps/api/webviewWindow";

async function toggleQuickCapture() {
  const existing = await WebviewWindow.getByLabel("capture");
  if (existing) {
    await existing.close();
    return;
  }

  const capture = new WebviewWindow("capture", {
    url: "/capture",          // React route for capture form
    title: "Quick Capture",
    width: 560,
    height: 280,
    center: true,
    decorations: false,       // Frameless
    transparent: true,        // For rounded corners
    alwaysOnTop: true,
    resizable: false,
    skipTaskbar: true,
    focus: true,
  });
}

// Register on app mount
await register("Control+Space", (event) => {
  if (event.state === "Pressed") {
    toggleQuickCapture();
  }
});
```

**Critical: The capture window needs its own HTML entry point or React router route.** Since the app currently has a single entry point, the simplest approach is to use a query parameter or hash route (`/capture`) and conditionally render the QuickCaptureWindow component.

### Pattern 5: Cross-Window Communication via Tauri Events

**What:** After the quick-capture window creates a task, it emits a Tauri event that the main window listens to for refreshing its task list.

```typescript
// In capture window after task creation:
import { emit } from "@tauri-apps/api/event";
await emit("task-created", newTask);

// In main window (already listening for task-created events):
// The existing event listener pattern handles this automatically
```

### Anti-Patterns to Avoid

- **Sharing Zustand store instances across windows:** Each Tauri window has its own JavaScript context. Do NOT try to share a Zustand store reference. Instead, use Tauri events for cross-window communication and let each window manage its own frontend state.
- **Using transparent: true on macOS without testing:** There are known Tauri issues with transparent frameless windows on macOS (border artifacts, packaging issues). Test early and have a fallback of `decorations: false` without `transparent: true` (use a solid dark background with border-radius in CSS instead).
- **Blocking the main thread with date calculations:** The time grouping and overdue detection should be simple comparisons, not expensive operations. Avoid creating Date objects in tight loops -- do string comparisons for date-only fields (`"2026-03-16" < "2026-03-17"` works correctly for ISO dates).

## Common Pitfalls

### Pitfall 1: SQLite ALTER TABLE Limitations
**What goes wrong:** SQLite ALTER TABLE cannot add columns with CHECK constraints in all versions. The `recurrence_rule CHECK(...)` constraint may fail on older SQLite versions.
**Why it happens:** SQLite bundled with rusqlite 0.32 is recent enough, but the syntax for CHECK on ALTER TABLE ADD COLUMN was only added in SQLite 3.31.0 (2020-01-22).
**How to avoid:** Verify rusqlite 0.32 bundles SQLite >= 3.31.0 (it bundles 3.45+). If unsure, add the CHECK constraint separately via a trigger or omit it and validate in Rust code.
**Warning signs:** Migration fails on app start with "near CHECK: syntax error."

### Pitfall 2: Row Index Fragility in row_to_task
**What goes wrong:** The current `row_to_task` function reads columns by positional index (0-9). Adding 5 new columns means updating every SELECT statement AND the index mapping.
**Why it happens:** The codebase uses positional indexing rather than named column access.
**How to avoid:** When modifying row_to_task, update ALL SELECT statements that feed into it. Consider defining a constant for the column list: `const TASK_COLUMNS: &str = "id, project_id, title, ...";`
**Warning signs:** Runtime panics with "column index out of range" or silently swapped field values.

### Pitfall 3: Quick-Capture Window Not Receiving Focus on macOS
**What goes wrong:** The frameless window appears but the title input does not receive keyboard focus.
**Why it happens:** macOS has strict rules about window focus. Frameless windows may not automatically become the key window.
**How to avoid:** Use `focus: true` in WebviewWindow options AND call `window.setFocus()` after creation. Add `autofocus` attribute to the title input element.
**Warning signs:** Window appears but typing goes to the previously focused application.

### Pitfall 4: Capability Permissions Missing for Quick-Capture Window
**What goes wrong:** Creating a WebviewWindow from JavaScript fails silently or throws a permission error.
**Why it happens:** Tauri 2's capability system requires explicit permission for window creation. The current `default.json` only has permissions for the "main" window.
**How to avoid:** Add `"core:webview:allow-create-webview-window"` to capabilities. Create a separate capability entry for the "capture" window with its required permissions (core:default, core:event, global-shortcut).
**Warning signs:** Console error about denied permission when calling `new WebviewWindow(...)`.

### Pitfall 5: Dual Type Systems for Task (types/task.ts vs lib/types.ts)
**What goes wrong:** The codebase has TWO TypeScript type definitions for Task -- `src/types/task.ts` (imported by useTaskStore) and `src/lib/types.ts` (imported by taskSlice and tauri.ts). They have different shapes and fields.
**Why it happens:** Different phases added types in different locations.
**How to avoid:** Consolidate scheduling fields into BOTH files, or better yet, pick one canonical location and re-export. The `src/lib/types.ts` file is the one used by the main store and API layer, so add scheduling fields there. Update `src/types/task.ts` to align or import from the canonical source.
**Warning signs:** TypeScript compilation succeeds but runtime data does not match type expectations because the wrong type was imported.

### Pitfall 6: Today View Query Performance
**What goes wrong:** The Today View needs tasks across ALL projects, with scheduling-aware filtering and sorting. A naive implementation queries all tasks and filters in JS.
**Why it happens:** The current `get_todays_tasks` already returns all non-complete tasks across all projects (no project filter). But it has no scheduling awareness.
**How to avoid:** Build the query in SQL:
```sql
SELECT ... FROM tasks
WHERE status != 'complete'
   OR (status = 'complete' AND scheduled_date = date('now'))
   OR (due_date IS NOT NULL AND due_date <= date('now'))
ORDER BY
  CASE WHEN due_date < date('now') AND status != 'complete' THEN 0 ELSE 1 END,
  scheduled_time ASC NULLS LAST,
  created_at DESC
```
**Warning signs:** Slow rendering when task count grows, or completed tasks from other days appearing.

### Today View Time Grouping

```typescript
interface TodayTask extends Task {
  projectName: string;  // Joined from projects table
}

type TimeGroup = "overdue" | "morning" | "afternoon" | "evening" | "unscheduled";

const TIME_GROUP_ORDER: TimeGroup[] = ["overdue", "morning", "afternoon", "evening", "unscheduled"];

function groupTasksByTime(tasks: TodayTask[], now: Date): Map<TimeGroup, TodayTask[]> {
  const today = now.toISOString().slice(0, 10);
  const groups = new Map<TimeGroup, TodayTask[]>();

  for (const group of TIME_GROUP_ORDER) {
    groups.set(group, []);
  }

  for (const task of tasks) {
    let group: TimeGroup = "unscheduled";

    if (task.dueDate && task.dueDate < today && task.status !== "complete") {
      group = "overdue";
    } else if (task.scheduledTime) {
      const hour = parseInt(task.scheduledTime.split(":")[0], 10);
      if (hour < 12) group = "morning";
      else if (hour < 17) group = "afternoon";
      else group = "evening";
    }

    groups.get(group)!.push(task);
  }

  return groups;
}
```

## Open Questions

1. **Transparent frameless window on macOS**
   - What we know: Tauri 2 supports `decorations: false` and `transparent: true`, but there are known macOS issues with border artifacts and packaging
   - What's unclear: Whether the combination works reliably in production builds on macOS
   - Recommendation: Try `decorations: false` without `transparent: true` first. Use CSS border-radius on the root element with a solid dark background. Only add `transparent: true` if rounded corners are essential.

2. **Quick-capture window routing**
   - What we know: The app uses a single entry point (`index.html`). The capture window needs to render only the QuickCaptureWindow component, not the full app layout.
   - What's unclear: Best approach for routing -- query param (`?window=capture`), hash route, or separate HTML entry point
   - Recommendation: Use a query parameter approach. In main.tsx, check `window.location.search` for `?window=capture` and render QuickCaptureWindow instead of the full App. Pass `url: "/capture"` or `url: "index.html?window=capture"` when creating the WebviewWindow. This avoids needing a separate Vite entry point.

3. **Capability configuration for the capture window**
   - What we know: Tauri 2 capabilities are scoped by window label. The current `default.json` targets `["main"]`.
   - What's unclear: Whether the capture window needs its own capability file or can share the default
   - Recommendation: Add `"capture"` to the `windows` array in `default.json`, since the capture window needs the same core permissions (invoke, events). Alternatively, create a minimal `capture.json` capability file.
