**Core function:** `pub fn compute_scores(db: &Database) -> Result<ScoringResult, String>`

Implementation approach (per Research Pitfall 5 -- avoid N+1):
- Single batch SQL query to get all active tasks with project info, due dates, statuses, and updated_at:
  ```sql
  SELECT t.id, t.title, t.status, t.due_date, t.updated_at, t.phase_id,
         p.id as project_id, p.name as project_name,
         ph.sort_order
  FROM tasks t
  LEFT JOIN projects p ON t.project_id = p.id
  LEFT JOIN phases ph ON t.phase_id = ph.id
  WHERE t.status != 'cancelled'
  AND (ph.sort_order IS NULL OR ph.sort_order < 999)
  ```
- Group tasks by project_id in Rust (HashMap)
- For each project, compute tags using `compute_project_tags()` helper
- For each project, compute priority_score: `1000.0 - days_to_soonest_deadline` (projects with no deadlines get score 0.0)
- Sort projects by priority_score descending
- Compute busy_score from today's scheduled_blocks and calendar_events:
  ```sql
  SELECT COUNT(*) as cnt, COALESCE(SUM(
    (CAST(substr(end_time,1,2) AS INTEGER)*60 + CAST(substr(end_time,4,2) AS INTEGER))
    - (CAST(substr(start_time,1,2) AS INTEGER)*60 + CAST(substr(start_time,4,2) AS INTEGER))
  ), 0) as total_minutes
  FROM scheduled_blocks WHERE date = ?
  ```
  And similarly for calendar_events. busy_score = committed_minutes / 480.0 (8-hour day), clamped to 0.0..1.0
- Count total_meetings from calendar_events for today
- Count total_tasks_due from tasks where due_date = today and status != complete

**Helper functions:**
- `fn compute_project_tags(tasks: &[TaskRow], today: NaiveDate) -> Vec<ProjectTag>` -- follows the exact logic from Research code example
- `fn is_recently_completed(updated_at: &str, today: NaiveDate) -> bool` -- task completed within last 24 hours (parse updated_at as NaiveDateTime, compare to today - 1 day)
- `fn format_deadline_item(title: &str, due_date: NaiveDate, today: NaiveDate) -> String` -- returns "Task X due tomorrow" or "Task X due in N days"
