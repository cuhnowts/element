This is the real concurrency bug identified per D-07. The `std::sync::Mutex` `TOKEN_REFRESH_LOCK` is held across `.await` points (lines 762-776), which can block the async runtime thread.

**Fix (3 changes in calendar.rs):**

1. Add import at top of file (near line 2):
```rust
use std::sync::LazyLock;
use tokio::sync::Mutex as AsyncMutex;
```

2. Replace the static declaration (line 742):
```rust
// Before:
static TOKEN_REFRESH_LOCK: Mutex<()> = Mutex::new(());

// After:
static TOKEN_REFRESH_LOCK: LazyLock<AsyncMutex<()>> = LazyLock::new(|| AsyncMutex::new(()));
```

3. Update the lock acquisition (lines 762-764):
```rust
// Before:
let _lock = TOKEN_REFRESH_LOCK
    .lock()
    .map_err(|e| CalendarError::OAuthError(e.to_string()))?;

// After:
let _lock = TOKEN_REFRESH_LOCK.lock().await;
```
Note: `tokio::sync::Mutex::lock()` returns the guard directly (no `Result`), so the `.map_err(...)` is removed entirely.

4. If the `use std::sync::Mutex;` import at line 2 is still needed by other code in calendar.rs (lines 750, 864, 885, 934 use `std::sync::Mutex` for Database and CredentialManager params), keep it. If those references use the full path `std::sync::Mutex`, the import may be unused — check and remove if clippy reports it.

**IMPORTANT:** Do NOT change any other Mutex in this file or in lib.rs. The `std::sync::Mutex<Database>` and `std::sync::Mutex<CredentialManager>` are NOT held across await points and are correct as-is (per Pitfall 5 from RESEARCH.md).
