## Interaction Contracts

### D-09: Risk Notification Click

- Risk notification appears in existing NotificationPopover using existing NotificationItem component
- Priority: `critical` for overdue or same-day risk, `informational` for multi-day-out risk
- `action_url` set to `hub://chat?context=risk&id={risk_id}` -- clicking opens hub chat with risk context pre-loaded
- Notification body contains the structured risk summary (e.g., "'Write report' due Thu has 4h work but only 2h available")
- Notification title format: "Deadline risk: {task_title}" for single task, "Deadline risks in {project_name}" for multiple tasks in same project

### D-11: Schedule Change Confirmation in Chat

- Bot sends a ScheduleChangeCard embedded in the chat message stream
- Card layout:
  - Header: 14px semibold, icon `CalendarClock` (lucide), title "Move Task"
  - Before line: 14px regular, `--color-muted-foreground`, strikethrough, e.g., "~~Today 2:00 - 4:00 PM~~"
  - After line: 14px regular, `--color-foreground`, e.g., "Tomorrow 9:00 - 11:00 AM"
  - Side effects (if any): 12px regular, `--color-muted-foreground`, e.g., "This pushes 'Code review' to tomorrow afternoon"
  - Separator
  - Action row: "Dismiss" (outline button, left) + "Confirm Move" (default button, right)
- Card uses `role="alertdialog"` with `aria-labelledby` and `aria-describedby` (same pattern as ActionConfirmCard)
- Keyboard: Enter confirms, Escape dismisses
- After resolution: card fades to 50% opacity, buttons replaced with "Confirmed" or "Dismissed" text (12px, muted-foreground)

### D-12: Risk Notification with Suggested Fix

- When a risk notification includes a suggested fix, the notification body appends: "Suggestion: {fix description}"
- Clicking the notification opens hub chat with the suggestion pre-loaded as a system context message (not visible to user as a chat bubble, but the bot references it in its first response)
- The bot then presents a ScheduleChangeCard for the suggested fix

### D-14: Suggest-Never-Auto-Apply

- All schedule changes require explicit user confirmation through ScheduleChangeCard
- No schedule modification happens without the user clicking "Confirm Move" or typing explicit approval in chat
- Bot never executes a reschedule tool call without first presenting the confirmation card

### Settings: Heartbeat Configuration

- New section in settings page: "Heartbeat" with 16px semibold heading
- Layout: vertical stack of fields with 16px (md) gap between fields
- Fields:
  1. **Enable heartbeat** -- Switch component, label "Background deadline monitoring", helper text "Check for deadline risks every {interval}" (12px, muted-foreground)
  2. **Check interval** -- Select with options: "15 minutes", "30 minutes" (default), "1 hour", "2 hours". Disabled when heartbeat switch is off
  3. **AI provider** -- Select with options populated from configured AI providers plus "None (template only)". Helper text "Used for risk summaries. Falls back to template when unavailable." (12px, muted-foreground)
- Status indicator: 6px circle dot to the right of the "Heartbeat" heading. Teal pulsing when active, muted static when inactive. `aria-label="Heartbeat is running"` or `"Heartbeat is stopped"`

## State Shape

New state needed for this phase:

```typescript
// heartbeatSlice (new Zustand slice)
interface HeartbeatState {
  heartbeatEnabled: boolean;       // persisted in settings DB
  heartbeatInterval: number;       // minutes, persisted in settings DB
  heartbeatProviderId: string | null; // AI provider ID, persisted in settings DB
  heartbeatRunning: boolean;       // runtime state from Tauri events
  lastHeartbeatAt: string | null;  // ISO timestamp of last check
  setHeartbeatEnabled: (enabled: boolean) => void;
  setHeartbeatInterval: (minutes: number) => void;
  setHeartbeatProviderId: (id: string | null) => void;
}
```

Integration with existing stores (read-only references, no new fields):
- `useNotificationStore` -- risk notifications flow through existing notification system
- `useBriefingStore` -- briefing content includes risk section when risks exist
- `useHubChatStore` -- schedule negotiation uses existing chat message stream

## Accessibility

| Requirement | Implementation |
|-------------|---------------|
| ScheduleChangeCard keyboard navigation | Enter confirms, Escape dismisses. Auto-focus on confirm button when card appears |
| ScheduleChangeCard screen reader | `role="alertdialog"`, `aria-labelledby` on title, `aria-describedby` on before/after body |
| Risk notification screen reader | Existing NotificationItem pattern with `role="button"` and keyboard handler |
| Heartbeat status dot | `aria-label` announces running/stopped state; not color-only (label text also available) |
| Settings switch | Existing shadcn Switch with associated label; disabled state on interval/provider communicated via `aria-disabled` |
| Strikethrough "before" text | Wrap in `<del>` element (not CSS-only strikethrough) so screen readers announce deletion |
| "After" text | Wrap in `<ins>` element so screen readers announce insertion |
